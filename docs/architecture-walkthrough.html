EKS SageMaker Triage

architecture walkthrough — ap-southeast-2

Infrastructure for an EKS-based agent that detects SageMaker pipeline failures via EventBridge, triages them with Bedrock (Kimi K2.5), and creates GitHub PRs proposing fixes. Built with Terraform, deployed via GitHub Actions with OIDC federation. Designed to be spun up and torn down on demand.

2 Bootstrap Layer

These resources were created outside of Terraform via the AWS CLI. They form a circular dependency: Terraform needs them to exist before it can store state, authenticate, or manage anything. Think of them as the foundation that Terraform stands on.

Why outside Terraform?

Terraform stores its state in S3 and locks it in DynamoDB. If those resources were inside Terraform, it would need them to exist before it could create them — a circular dependency. Similarly, GitHub Actions authenticates via the OIDC provider to run Terraform, so the provider must exist first.

What was created
  • terraform-admin IAM user with AdministratorAccess
  • S3 bucket: eks-sagemaker-triage-tfstate-208107893626
  • DynamoDB table: eks-sagemaker-triage-tflock
  • GitHub OIDC provider + GitHubActionsRole
  • CloudWatch alarms at $20 and $30
3 Terraform Foundation

Three files form the base every other file depends on. backend.tf tells Terraform where to store state. providers.tf configures AWS with default tags. variables.tf centralises tuneable parameters with sensible defaults so terraform apply works without a .tfvars file.

backend.tf

Remote State

S3 backend stores terraform.tfstate remotely. DynamoDB table acts as a mutex lock — prevents two people from applying simultaneously. encrypt = true encrypts state at rest.

providers.tf

AWS Provider

Pins AWS provider to ~> 5.0. Sets default_tags so every resource inherits Project and ManagedBy tags. Defines data sources: aws_caller_identity, aws_region, aws_availability_zones.

variables.tf

Input Variables

Centralises: aws_region (ap-southeast-2), project_name, account_id, kubernetes_version (1.31), and common_tags. All have defaults so no .tfvars needed.

Data sources are read-only lookups. aws_caller_identity fetches your account ID so you don't hardcode it. aws_availability_zones discovers which AZs are available for subnet placement. They don't create resources.
4 Networking

The VPC is your private network within AWS. Everything runs in public subnets only — no NAT gateway — saving ~$32/month. Nodes get public IPs and access the internet directly through an internet gateway.

Why public subnets?

A NAT gateway costs ~$32/month + data processing. In production, nodes sit in private subnets (no public IPs) and use NAT for outbound internet. For a learning project, public subnets are fine — the security trade-off is acceptable and saves real money.

Why two AZs?

EKS requires subnets in at least 2 availability zones for high availability. Each AZ is a separate physical data centre. Even though we only run one node, EKS enforces this constraint. The kubernetes.io/role/elb tags are for future load balancer support.

5 EKS Cluster

EKS has two distinct parts that cost differently. The control plane (API server, etcd, scheduler) is a managed service AWS runs outside your VPC — $0.10/hr. The node group is the EC2 instance where pods actually run — $0.026/hr for t3.small.

Control Plane

Managed by AWS

Kubernetes API server, etcd, scheduler, controller manager. Lives outside your VPC. You just get an API endpoint.

  • Cost: ~$0.10/hr ($73/month)
  • Public endpoint for kubectl from local
  • Logging: audit + api to CloudWatch
  • depends_on ensures IAM role exists before cluster creation
Managed Node Group

Your compute

EC2 instances where pods run. "Managed" means AWS handles provisioning, updates, and draining.

  • Single t3.small: 2 vCPU, 2 GiB RAM
  • Cost: ~$0.026/hr ($19/month)
  • Scaling: min=1, max=1, desired=1 (no autoscaling surprises)
  • ON_DEMAND (not Spot — simpler, no interruptions)
  • 20 GiB root volume
Cost warning. The EKS control plane alone costs $73/month. Combined with the node, you're looking at ~$92/month if left running. This is why terraform destroy when done is critical — and why we set up the auto-destroy hook.
6 Container Registry

ECR stores the Docker image for your triage agent. When you build the agent later, you docker push to ECR, and the EKS node docker pulls it to run the pod. The node can pull because its IAM role has AmazonEC2ContainerRegistryReadOnly.

Mutable Tags

Lets you push the same tag (like :latest) repeatedly. In production you'd use IMMUTABLE to enforce unique tags, but for dev it speeds up iteration.

Scan on Push

Automatically checks each image for known CVEs (security vulnerabilities). Free and useful — gives you a security baseline without extra tooling.

Lifecycle Policy

Automatically deletes images beyond the 5 most recent. Without this, every docker push accumulates storage costs forever.

8 IAM Roles — The Permission Model

Every AWS service interaction requires explicit permission. Each role has a trust policy (who can assume it) and permissions policies (what it can do). The file defines 5 roles.

Role Trusted By Permissions Purpose
eks-cluster-role eks.amazonaws.com AmazonEKSClusterPolicy EKS service manages cluster networking and compute
eks-node-role ec2.amazonaws.com WorkerNode + CNI + ECR Read Nodes register with cluster, manage ENIs, pull images
triage-agent-pod-role EKS OIDC Provider (IRSA) Bedrock invoke + SageMaker read + Secrets read Pod-level permissions scoped via service account
sagemaker-pipeline-role sagemaker.amazonaws.com Pipeline exec + CloudWatch Logs + PassRole Execute the dummy fail pipeline
eventbridge-target-role events.amazonaws.com CloudWatch Logs (stub) Placeholder — will expand for Lambda/EKS target
Understanding IRSA (IAM Roles for Service Accounts)

Without IRSA, every pod on a node inherits the node's IAM role — too permissive. IRSA lets you grant specific permissions to specific pods.

How it works:

1. EKS creates an OIDC provider (same concept as GitHub Actions federation). 2. You create an IAM role whose trust policy says "only the K8s service account triage-agent in namespace default can assume this role." 3. You annotate the ServiceAccount with the role ARN. 4. The K8s webhook injects temporary AWS credentials scoped to that role.

The triage agent pod gets three narrowly scoped permissions:

• Bedrock — invoke moonshotai.kimi-k2.5 only (not other models)
• SageMaker — read pipeline definitions, executions, and logs (not create/modify)
• Secrets Manager — read secrets prefixed with eks-sagemaker-triage/

9 CI/CD Pipeline

Two GitHub Actions workflows enforce the principle: no infrastructure changes without code review. PRs get a plan, merges get an apply. Authentication uses OIDC — no stored AWS keys.

How OIDC federation works (no stored secrets)

1. You told AWS "I trust GitHub" by creating an OIDC identity provider.

2. You created GitHubActionsRole with a trust policy scoped to repo:lmwc93/eks-sagemaker-triage-infra:*.

3. When a workflow runs, GitHub generates a short-lived JWT proving "I am a workflow in this repo on this branch."

4. The aws-actions/configure-aws-credentials action presents this JWT to AWS STS.

5. AWS validates the token against the OIDC provider, confirms the repo matches, and issues temporary credentials that expire automatically.

Result: no stored secrets, no key rotation, credentials scoped to your specific repo. The workflow needs permissions: id-token: write for this to work.

10 Developer Experience

The .claude/settings.json file configures Claude Code with hooks and an MCP server, all scoped to this project only.

SessionStart Hook

Reminder

Prints a warning on every session start: "exit with /exit, not the red circle." Fires on all sessions in this project.

SessionEnd Hook

Auto Destroy

Runs terraform destroy -auto-approve when you type /exit. Matcher: "logout" only — does NOT fire on /clear or other exits. Logs to /tmp/terraform-destroy.log.

Context7 MCP

Live Docs

MCP server that provides up-to-date documentation for AWS services. On-demand (2 tools, ~5K tokens per query). Prevents stale training data issues.

Caveat. The SessionEnd hook only fires on graceful exits. If you force-quit the terminal (Cmd+Q, kill -9, laptop dies), the hook won't run and your cluster stays up accruing cost. The CloudWatch alarms at $20 and $30 are your backstop.

12 Cost Summary

The critical insight: most costs only accrue when the cluster is running. Tearing down with terraform destroy eliminates the expensive resources. The table below shows estimated monthly costs.

Resource Running Destroyed Notes
EKS Control Plane $73.00 $0.00 $0.10/hr, biggest fixed cost
EC2 Node (t3.small) $19.00 $0.00 $0.026/hr, on-demand
CloudWatch Logs ~$2.00 $0.00 Control plane audit + api logs
ECR Storage ~$0.10 $0.00 5 images max, lifecycle policy
S3 State Bucket ~$0.01 ~$0.01 Always exists (bootstrap), negligible
DynamoDB Lock Table $0.00 $0.00 Pay-per-request, near-zero usage
Bedrock (Kimi K2.5) ~$0.05 $0.00 $0.60/M input tokens, pay-per-use
Total (estimated) ~$94/month ~$0.01/month Destroy when not using!
Key takeaway. At ~$3/day when running, remember to terraform destroy after each session. The auto-destroy hook on /exit handles this, but the CloudWatch alarms at $20 and $30 are your safety net if anything slips through.
